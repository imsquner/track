## 3. 传感器状态位与控制逻辑

### 3.1 五路红外状态编码
LightSensor_GetState 返回 8 位状态字（实际使用低 5 位）：
- bit4 = L1(PB4)
- bit3 = L2(PB3)
- bit2 = M(PB15)
- bit1 = R2(PB14)
- bit0 = R1(PB13)

在 Track_Run 中，逻辑按 1=白底、0=黑线处理。

### 3.2 典型循迹决策
- 0x04（中间压线）直行
- 0x08/0x10（偏左）右转
- 0x02/0x01（偏右）左转
- 0x0C/0x18（左双触发）右转
- 0x06/0x03（右双触发）左转
- default 停车

## 4. 串口对应关系与现状

### 4.1 当前工程串口现状
- 目前没有任何 USART/UART 初始化、发送、接收代码。
- 即：当前版本串口功能未启用。

### 4.2 可用串口建议（后续调试最实用）
建议优先启用 USART1：
- TX: PA9
- RX: PA10

用途：
- 输出五路红外原始状态
- 输出速度与转向决策
- 调参数（例如直行速度、转向速度）

## 5. 根本性设计点（你项目里最关键的底层事实）

1. 传感器接口模式
- 你现在使用的是 DO 数字输出方案，因此每路只需要 1 个 IO。
- 代码实际配置为输入上拉，不是浮空输入。

2. JTAG 与循迹引脚冲突
- PB3/PB4 默认会被调试口占用。
- 你已在 LightSensor_Init 中关闭 JTAG（保留 SWD）来释放引脚，这是必须步骤。

3. 电机控制是“方向 + PWM”双通道模型
- 每个电机两路方向脚 + 一路 PWM。
- 速度符号决定方向，绝对值决定占空比。

4. PWM 频率基础
- TIM2 预分频 36、重装值 100。
- 若定时器时钟 72MHz，PWM 频率约 20kHz，属于电机可用频段。

5. 当前工程存在引脚复用冲突风险
- PA1/PA2 同时被 LED 与电机 PWM 使用。
- 实际运行中，若 LED_Init 与 Motor_Init 都调用，会发生功能冲突。

6. 串口缺位导致可观测性不足
- 目前只能靠 OLED 看状态，调试效率低。
- 增加串口日志后，排查传感器抖动、阈值问题会快很多。

## 6. 针对你笔记中“IO 不够”问题的直接方案

1. 维持当前做法（推荐起步）
- 五路红外全部用 DO 数字口。
- 优点：实现简单，实时性好。

2. 若后续 IO 仍紧张
- 方案A：加 I2C IO 扩展（如 PCF8574）
- 方案B：加并入串出芯片（如 74HC165）
- 方案C：使用 AO + ADC + 多路复用器做灰度采样（复杂度最高）

3. 工程优先级建议
- 先把串口调试口加上
- 再消除 PA1/PA2 与 LED 的冲突
- 最后再考虑 IO 扩展

## 7. 一句话结论

你的循迹主链路已经成型：五路红外输入 -> 状态编码 -> 规则决策 -> 双电机 PWM 驱动。当前最影响进度的根本问题不是算法，而是“可观测性不足（无串口）+ 引脚复用冲突（PA1/PA2）”。
